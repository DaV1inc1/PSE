Job Options

The following graphs describe the options how a job cycle is defined. All options can interoperate. The execute() function is called in the beginning of each job cycle for Cyclic jobs.

Job Synchronization

The job synchronization is done by the scheduler. The relevant options are stored in the job itself. The options are:

  • Syncronization: the flag is provided with the constructor
  • TimeoutTime: provided with setTimeoutTime()

Cyclic job

Default

In this example every 100 cycles execute() is called and the current buffered message is used. The message is updated with prepare(). This means that the execute() command in cycle 200 does exactly the same as the command in cycle 100, because the message didn't get updated with a prepare() function.

  • cycletime = 100

Syncronized

The scheduler waits, that each job cycle gets prepared. The prior setup wouldn't be possible, because job cycle 2 doesn't have a message input, be the prepare function. Here at the end of job cycle 2 the lock function from the scheduler waits until the job gets prepared.

  • cycletime = 100
  • Syncronization = true
  • TimeoutTime() = 5

Trigger job

Default

The default trigger job isn't bound to the job cycles. As the name says, it gets executed, when triggered. The trigger is realized with the prepare() function.

  • cycletime = 100

Synchronized

Each job cycle needs a perpare() trigger. Otherwise the scheduler halts the simulation in the current cycle and waits for the trigger.

  • cycletime = 100
  • Syncronization = true
  • TimeoutTime() = 5
