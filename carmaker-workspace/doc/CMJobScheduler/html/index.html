JobScheduler Documentation

The library provides the user with a job and a scheduler interface.

Job

The provided interface is an abstract C++ class. If the user wants to create a specific job, f.e. ROS subscriber, TCP/IP sender, ... . One has to create a new class and inherite CMJob::AbstractJob.
The user has to define the function:

  • execute()

The execution function is shown in the statemachine. As example look into CMJob_RosSubscriber.h.

The user can define following functions:

  • init()
  • prepare()
  • activate()
  • suspend()
  • disable()

Each job is based on following statemachine:

There are 2 different types:

  • Cyclic Job: This job is executed in a fixed cycletime and in the beginning of the job cycle.
  • Trigger Job: This job is executed, when it gets triggered.

Options

  • Syncronization: the flag is provided with the constructor. When it is active, the job needs to get prepared. Otherwise it goes into Suspended state and finally into Timeout state, when it is due.
  • CycleTime: provided with setCycleTime() are the cycles when the job is due.
  • CycleOffset: provided with setCycleOffset() is the offset within a cycles when the job is due.
  • SkipFirstCycles: provided with skipFirstCycles() are the first cycles when the job is skipped.
  • TimeoutTime: provided with setTimeoutTime() is the walltime, how long the job can stay in Suspended state, before it gets put into Timeout state
  • CallbackHook: provided with setCallbackHook() used by the scheduler, to identify when the job should be executed.
  • ExecutionCounter: provided with setExecutionCounter() how often the job should get executed, before it gets disabled

Please look at the examples for a more detailed explanation of the options.

Scheduler

The scheduler is an interface to control job instances. It screens all registered jobs, if they are ready and when their callback has to be executed.

The user has to integrate following calls into his code:

  • resetJobs(): Call it before the simulation starts.
  • startCycle(): Prepares jobs in the beginning of each cycle. Call it in the beginning of the cycle.
  • lock(): Waits for suspended jobs, in the beginning of each cycle. Call it after startCycle.
  • endCycle(): Update jobs at the end of each cycle. Call it at the end of the cycle.
  • executeJobs(): For each hook, this function has to be placed in code
  • disableJobs(): Call at the end of simulation
